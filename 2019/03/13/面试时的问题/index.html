Activity launchMode (启动模式)

任务和返回堆栈

standard

默认模式，可多次实例化。每个实例可属于不同的任务，一个任务可拥有多个实例。

适用场景：大多数应用场景

singleTop

栈顶复用模式。若堆栈栈顶已经是当前实例，则获取已有实例；否则创建新的实例。

适用场景：通知消息打开的页面。

反面教材：微博

正面教材：QQ的空间和微博国际版

singleTask

栈内复用模式。检索栈内所有实例，若存在对应实例，则获取已有实例，并将之上的实例出栈；不存在，则新建实例。

适用场景：大多数APP的主页。

singleInstance

单一实例模式。实例会单独占用一个栈，在系统中具有全局唯一性。此模式的检索范围扩大至整个系统。

适用场景：呼叫来电、闹钟响铃等界面。

内存泄露与内存优化

Android性能优化之从卡顿和ANR来彻底理解内存泄露原理和优化

使用 Android Studio 的 Memory Monitor 或 LeakCanary 可查看及检测内存泄露。

本质原因

持有对象引用。持有引用者的生命周期超过了被引用者的生命周期

常见内存泄露场景

  • 单例模式持有某个 Activity 或者是 service 上下文的引用，导致不能正常回收

可以将 context 改为全局的上下文

  • 静态变量持有 Activity 引用导致内存泄露

尽量减少该情况；另外要适当的时候重置为 null。

  • 非静态内部类（包括匿名内部类）导致泄露

比如 Handler 的使用，一般要改成静态内部类加弱引用的方式。

  • 未及时取消注册一些组件，比如广播等

  • Timer 和 TimerTask 导致泄露

常用来做一些无限轮播的 ViewPager 等，销毁 Activity 时未能及时 cancel。

  • 集合中的对象未能及时清理

  • 进行读写操作时资源未关闭

在使用 IO、File、Sqlite 等读写操作时要及时关闭

  • 属性动画未及时 cancel

  • WebView

WebView 的 Callback 常持有 Activity 引用，造成即使调用 WebView.destroy() 等方法也无法释放。正确的方案是需要先将 WebView 从父容器移除，再进行销毁。

调起支付宝支付的流程

https 加密

客户端向服务器端发起第一次请求，服务器将自己的一对秘钥和公钥中的公钥发送给客户端，然后客户端对公钥进行验证。验证合格后，客户端生成一个秘钥，然后用服务器的公钥对这个秘钥进行加密。

客户端发起第二次请求，将加密过后的秘钥发送过去，服务器用自己的私钥进行解密，解密之后得到客户端秘钥，然后利用客户端秘钥对数据进行加密，将加密后的数据发送给客户端。客户端得到数据后，用秘钥再进行解密，得到数据。

类与接口

类可以同时继承两个吗？有什么方法可以继承两个？接口里有变量吗？

类

单继承

Java 只允许一个class继承自一个类，即可有且仅有一个父类。除 Object 特殊，没有父类。

子类无法访问父类的 private 字段或 private 方法。为了访问，可以将 private 修改为 protected。

在构造方法中，系统会自动隐式调用父类的无参构造函数。如果父类没有默认的构造方法，子类必须显式调用 super() 并给出参数以便定位到父类的构造方法。

阻止继承

final 关键字修饰的变量的值不可以更改；final 关键字修饰的类不可被继承；final 关键字修饰的方法可以被继承，但不能被子类重写。

Java 新技术：封闭类

Java 15 开始，也可使用 sealed 修饰 class，并通过 permits 明确写出能够继承该 class 的子类名称。

继承它的类必须声明为 终极类(final)、封闭类(sealed) 或解封类 (non-sealed) 其中一种。

final

final 修饰的类不可被继承；final 修饰的方法不可被重写；final 修饰的变量一经初始化后不可更改。

static

static 修饰的变量为静态变量，被所有的对象所共享，在内存中只有一个副本，当且仅当在类初次加载时被初始化一次。

static 修饰的方法为静态方法，不依赖于任何对象即可调用。静态方法中不可访问非静态成员方法和非静态成员变量。

static 还可用来形成静态代码块，可多个置于类中的不同地方，初次加载时按照顺序依次加载且仅加载此次一次。

abstract

abstract 可以修饰类和方法，修饰类为抽象类，修饰方法为抽象方法。只有接口和类中才能有抽象方法。

抽象方法被子类继承后必须重写。已实现的方法可选择性重写。

抽象方法仅可被 public、protected 两种修饰，若不为抽象方法则不限制。

abstract 不可与 final 和 private 共存，因为前者要求继承实现，后两者修饰后不可被继承，相矛盾冲突。

abstract 不可与 static 共存，因为抽象方法还未实现，static 修饰的方法可被外部直接调用，相矛盾冲突。

但抽象类中是可以有 private 或 final 修饰的方法的，只不过无法被外部调用或继承重写。

RelativeLayout 与 LinearLayout 的区别

布局上

LinearLayout 会默认保持组件之间的间隔以及相互对齐。可通过 orientation 来设计布局的方向，垂直和水平。

RelativeLayout 相对线性来说比较灵活。线性布局往往需要布局嵌套，想相对布局可以指定与其他组件或父组件的位置。

性能上

LinearLayout和RelativeLayout性能区别

  • RelativeLayout 慢于 LinearLayout 是因为它会让子 View 调用 2 次 measure 过程，而正常情况下 LinearLayout 只需一次。但若存在 weight 属性，LinearLayout 也需 2 次 measure，先避开设置 weight 属性的 view 做一次 measure，再对设置过 weight 属性的 view 做一次 measure
  • 在不考虑层级深度的情况下，使用 LinearLayout 而非 RelativeLayout

总结，层级多用 RelativeLayout ，层级固定用 LinearLayout

自定义 View 事件机制

自定义 View 事件分发机制

前言

MotionEvent 事件

手指触碰屏幕后，产生以下几种事件：

  • ACTION_DOWN：手指接触屏幕
  • ACTION_MOVE：手指在屏幕上移动
  • ACTION_UP：手指松开屏幕

产生的一系列事件

  • 点击屏幕后松开，序列为：DOWN -> UP
  • 点击屏幕滑动一会松开，序列为：DOWN -> MOVE -> MOVE -> … -> MOVE -> UP

通过 MotionEvent 事件获取 x，y 坐标

  • getX()/getY() 方法获取相当于 View 左上角的 x，y 坐标
  • getRawX()/getRawY() 方法获取相当于手机屏幕左上角的 x，y 坐标

View 的事件分发机制

事件传递顺序

如果子 View 没有处理事件，则事件没有被消耗，会反向传递上来

事件分发的三大方法

ViewGroup 中有 3 个跟事件分发有关的方法，分别是 dispatchTouchEvent、onInterceptTouchEvent、onTouchEvent。

dispatchTouchEvent 方法

onInterceptTouchEvent 方法

onTouchEvent 方法

onTouchListener、onTouchEvent、onClickListenre 的优先级

1、onTouchListener 和 onTouchEvent 都在 dispatchTouchEvent 方法中被调用，onClickListener 在 onTouchEvent 方法中被调用

2、onTouchListener 的优先级高于 onTouchEvent 方法，如果 onTouchListener 的 onTouch 方法返回 true，则 onTouchEvent 方法不会被调用，当然 onClickListener 就更不会被调用了

3、在 onTouchEvent 方法中，如果当前 View 设置了 onClickListener，那么 onClickListener 的 onClick 方法会被调用

4、只要 View 的 Clickable 和 Longkg_Clickable 有一个为 true，View 就会消耗当前事件，也就是 onTouchEvent 方法最后会返回 true

5、View 的 Long_Clickable 属性默认为 false；可点击的 View 的 Clickable 属性为 true，不可点击的 View 的 Clickable 属性为 false

事件分发流程

1、事件传递到 ViewGroup 时，dispatchTouchEvent 方法会被调用。如果这个 ViewGroup 的 onInterceptTouchEvent 方法返回 true，则表示它要拦截事件，事件会交给当前 ViewGroup 的 onTouchEvent 方法处理

2、如果当时 ViewGroup 的 onInterceptTouchEvent 返回 false，即不拦截事件，则会调用子元素的 dispatchTouchEvent 方法，这样就把事件传递给了子元素

3、如果子元素没有消耗事件，也就是子元素的 dispatchTouchEvent 方法返回 false，那事件会由当前 ViewGroup 自己处理，当前 ViewGroup 的 onTouchEvent 会被调用。如果当前 ViewGroup 的 dispatchTouchEvent 方法也返回 false，最后会一层层往上返回。如果事件一直没被消耗，那么最后 Activity 的 onTouchEvent 方法会被调用。

4、ViewGroup 中并没有 onTouchEven t方法。在所有子元素没有消耗事件时，ViewGroup 会调用父类，也就是 View 的 dispatchTouchEvent 方法，从而调用到 onTouchEvent 方法来自己处理事件，如果自己没有消耗事件，dispatchTouchEvent 方法就会返回 false，从而将事件反向往上层传递。

5、如果 ACTION_DOWN 事件子元素没处理(onTouchEvent 返回 false)，那这个事件序列的其他事件(MOVE 和 UP 事件)都不会再分派给子元素处理。

事件分发逻辑

1、View 中没有 onInterceptTouchEvent 方法，所以一旦事件传递到 View，那么 View 的 dispatchTouchEvent 方法就会被调用。

2、dispatchTouchEvent 方法中处理事件的逻辑顺序是 onTouchListener —> onTouchEvent —> onClickListener。

3、如果 View 设置了 onTouchListener，那 onTouchListene 的 onTouch 方法会被调用。如果 onTouch 方法返回 true，那事件就被消耗了，事件分发结束，onTouchEvent 不会被调用。

4、如果 onTouch 方法返回 false，那么 onTouchEvent 就会被调用。如果 View 设置了 onClickListener，当 ACTION_UP 事件到来时，onTouchEvent 中的onClickListener 的 onClick 方法也会被调用。

5、View 一般都会消耗事件，如果 View 没有消耗 ACTION_DOWN 事件，那后面 ACTION_MOVE 和 ACTION_UP 就都不会传递给 View。
